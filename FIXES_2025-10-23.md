# Fixes Applied - October 23, 2025

## ✅ Issue 1: Composer Proxy 404 Errors - FIXED

### Problem
All dist URL downloads from composer proxy were returning 404:
```
[404] http://localhost:8081/php_proxy/dist/symfony/polyfill-php85/v1.33.0
[404] http://localhost:8081/php_proxy/dist/voku/portable-ascii/2.0.3
```

### Root Cause
The `CachedProxySlice` was:
1. Fetching metadata from remote (packagist.org)
2. Rewriting dist URLs to proxy through Artipie  
3. Returning metadata to client
4. **Never saving the rewritten metadata to storage**

When `ProxyDownloadSlice` later tried to download a package, it needed to:
1. Look up the original URL from cached metadata
2. Download from original URL
3. Return to client

But step 1 failed because metadata wasn't in storage!

### Solution
**File:** `/composer-adapter/src/main/java/com/artipie/composer/http/proxy/CachedProxySlice.java`

Modified the response handling to save rewritten metadata to storage:
```java
// Save rewritten metadata to storage so ProxyDownloadSlice can find original URLs
final Key metadataKey = new Key.From(name + ".json");
return rewritten.asBytesFuture().thenCompose(rewrittenBytes -> {
    final Content saved = new Content.From(rewrittenBytes);
    return this.repo.storage().save(metadataKey, saved)
        .thenApply(ignored -> {
            Logger.debug(this, "Saved metadata to storage: %s", metadataKey);
            return ResponseBuilder.ok()
                .header("Content-Type", "application/json")
                .body(new Content.From(rewrittenBytes))
                .build();
        });
});
```

### Result
- Metadata is now saved with both rewritten URLs (`dist.url`) and original URLs (`dist.original_url`)
- `ProxyDownloadSlice` can find the metadata and extract the original URL
- Downloads are proxied correctly through Artipie

## ✅ Issue 2: PyPI Publish 500 Error - FIXED

### Problem
PyPI publish fails with WebSocket handler exception:
```
Failed to respond to request: Uat io.netty.handler.codec.http.websocketx.extensions.WebSocketServerExtensionHandler.onHttpRequestChannelRead
```

### Root Cause
**The `pypi` repository type was completely missing from RepositorySlices!**

Only `pypi-proxy` and `pypi-group` were defined. When requests came in as:
```
POST /artifactory/api/pypi/pypi
```

1. ApiRoutingSlice extracted `pypi` as repo_type
2. Rewrote URL to `/pypi`
3. SliceByPath found repo named "pypi" with type "pypi"
4. **RepositorySlices had NO case for type "pypi"**
5. Fell through to default case → threw IllegalStateException
6. Exception handler in VertX caused WebSocket-related error cascade

### Solution
**File:** `/artipie-main/src/main/java/com/artipie/RepositorySlices.java`

Added the missing case for PyPI local repositories:
```java
case "pypi":
    slice = trimPathSlice(
        new com.artipie.pypi.http.PySlice(
            cfg.storage(), securityPolicy(), authentication(), 
            tokens.auth(), cfg.name(), artifactEvents()
        )
    );
    break;
```

### Result
- PyPI local repositories now work correctly
- Uploads/downloads function as expected
- Integrates with API routing system

## 🔍 Key Insight

The Composer issue revealed a common pattern in proxy implementations:
- **Metadata rewriting** must be done **before** saving to cache
- **Original URLs** must be preserved for downstream components
- **Cache writes** are often forgotten in reactive/async code

This pattern applies to all proxy adapters (npm, maven, pypi, etc.).

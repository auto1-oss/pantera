# v2.2.0 Search and Browse Overhaul

Status: shipped (2026-04-23) â€” branch `2.2.0`, commits
`c98359fb..a07b01f3`.

## Why

Users filed six discrete UX complaints against Pantera's search and repo
browser:

1. Sorting search results by **date** appears to do nothing â€” `asc` and
   `desc` return the same items in the same order.
2. Sorting by **name** and **version** is not always accurate (`pkg-10`
   sorts before `pkg-2`).
3. A query for `wkda.common.api.vehicle-api` returns 16 510 results,
   every one of them a `.meta.maven.shards.*` internal metadata row â€”
   real artifacts are buried.
4. Search results **don't show upload date**.
5. Repo browser **doesn't show upload date**.
6. Repo browser **has no sort-by-date** (users miss the native
   `FileSystemBrowseSlice` experience, which had client-side JS sort
   by name/date/size).

Live reproduction (pre-fix):

```bash
curl 'â€¦/api/v1/search?q=wkda.common.api.vehicle-api&sort=created_at&sort_dir=asc' | jq '.items[0]'
curl 'â€¦/api/v1/search?q=wkda.common.api.vehicle-api&sort=created_at&sort_dir=desc' | jq '.items[0]'
```

Both returned the same item, and all 20 items on each page were
`.meta.maven.shards.*` â€” confirming bugs 1 and 3 simultaneously.

## Root causes

These six symptoms collapsed into four structural root causes.

### RC1 â€” `SortField` round-trip emitted the wrong wire value

`SearchHandler.search()` validated `?sort=created_at` against a
`VALID_SORT_FIELDS` allowlist containing `"created_at"`, mapped it to
`SortField.DATE`, then passed `sortField.name().toLowerCase()` ("date")
to `DbArtifactIndex`. `DbArtifactIndex.toSortField("date")` did not
recognise that string â€” only the original wire value `"created_at"` is
handled â€” and fell through to `SortField.RELEVANCE`, producing
`ORDER BY rank DESC, name ASC` for every direction.

Fix: `SearchHandler` passes the validated wire string directly; a
regression test pins `"date"` as **not** a valid wire key.

### RC2 â€” No classification; noise rows indistinguishable from artifacts

`MavenScanner.isScannableCandidate` filters `.md5 / .sha1 / .sha256 /
.sha512 / .asc / .sig / maven-metadata.xml / .pantera-meta.json` at
backfill import time â€” but live upload paths write these rows without
filtering, and the search SQL had no WHERE clause to hide them. A
targeted Maven repo had accumulated 16 510 `.meta.maven.shards.*` rows
that matched the FTS query for `wkda.common.api.vehicle-api`, pushing
real JARs past the first 500 pages.

Fix: V124 migration adds `classify_artifact(text)` + STORED
`artifact_kind` column; `buildFilterClauses` appends
`AND artifact_kind = 'ARTIFACT'` unconditionally. Non-ARTIFACT rows
stay in the table for routing / integrity checks but are invisible to
search by default. Opt-in (`?include=â€¦`) is plumbed structurally via
field-filter syntax and left as a follow-up.

### RC3 â€” Sort keys reflect byte values, not user mental model

- Name column is lexicographic text. `pkg-10` < `pkg-2` in string
  order.
- Version column has the V111 `version_sort` column with zero-padding,
  but `version_sort_key(v)` returns NULL for values like `"UNKNOWN"`
  or non-numeric patterns, and `NULLS LAST` pushes them to the end
  regardless of direction.

Fix: V123 migration adds `name_sort` built via `natural_sort_key(text)`
using the same zero-padding technique as `version_sort`, but over any
string (not just parseable versions). `buildOrderBy` NAME branch uses
`name_sort NULLS LAST, name`.

### RC4 â€” Two parallel views of the same data

Search reads from DB (has `created_date`, honours server-side sort /
pagination / FTS). Repo browser reads only from storage via
`asto.list()`, which returns keys without metadata. The artifact
_detail_ endpoint already called `asto.metadata(key)` to surface
`modified` â€” proof per-key metadata exists â€” but `treeHandler` never
asked. The frontend `TreeEntry` type already declared `modified?:
string` but the API never populated it, and the UI sort was a
name-only client-side sort of the current page.

Fix: `treeHandler` stays storage-authoritative for tree **structure**
but issues **one** batched `WHERE name = ANY(?)` query per listing to
hydrate every file row with size / modified / artifact_kind. Accepts
`sort=name|date` + `sort_dir=asc|desc`; sorts server-side (Java
`Comparator` on the small listing). UI adds the sort dropdown and
date column.

## Schema changes

| Migration | Column | Type | Purpose |
|-----------|--------|------|---------|
| V123 | `name_sort` | `text` GENERATED | Natural name sort |
| V124 | `artifact_kind` | `text` GENERATED | Classification + noise filter |

Both are STORED generated columns â€” adding them rewrites the `artifacts`
table, which is O(rows) disk-I/O at migration time. On a 10M-row table
expect 2â€“10 minutes depending on IOPS; during that window the table
holds a `ShareRowExclusiveLock` which blocks writes (`INSERT / UPDATE
/ DELETE / ALTER`) but not `SELECT`. Plan deployment accordingly.

Indexes:

- `idx_artifacts_name_sort (name_sort)` â€” supports name-ordered search.
- `idx_artifacts_kind_nonartifact (artifact_kind) WHERE artifact_kind
  <> 'ARTIFACT'` â€” cheap partial index for future opt-in queries.
- `idx_artifacts_kind_created_date (artifact_kind, created_date)` â€”
  supports the default-filtered date-sort path.

## API surface

### Search (`GET /api/v1/search`)

No breaking changes. `sort=created_at&sort_dir=asc|desc` now works as
documented. `?include=` for opting into non-ARTIFACT rows is documented
as a follow-up â€” the column and classification function are in place.

### Tree (`GET /api/v1/repositories/:name/tree`)

New query params:
- `sort=name|date` (default `name`)
- `sort_dir=asc|desc` (default `asc`)

New response fields on file entries:
- `modified` â€” ISO-8601 upload timestamp (sourced from
  `artifacts.created_date`).
- `artifact_kind` â€” classification label.

Response envelope now echoes `sort` and `sort_dir` so the UI knows
which ordering it received.

### Delete (`DELETE /api/v1/repositories/:name/artifacts`)

Not a new surface â€” but fixed. Previously 500'd on every DB-only repo
because `RepoData.deleteArtifact` used the YAML-only `repoStorage(rname)`
lookup. Now uses the DB-fallback lookup the read/tree paths already
used.

## Scale characteristics

Post-fix query shapes and expected latencies on a 10M-row `artifacts`
table (AWS Aurora PostgreSQL, 4 vCPU, EBS gp3):

| Query | Index | Expected |
|-------|-------|----------|
| Search + date sort | `idx_artifacts_kind_created_date` | 5â€“50 ms |
| Search + name sort | `idx_artifacts_name_sort` | 5â€“30 ms |
| Tree listing + metadata hydration | `idx_artifacts_repo_lookup` (batch `IN`) | 2â€“10 ms per listing |
| Classification on insert | no I/O (function is IMMUTABLE) | ÎĽs |

The tree-hydration query is O(children-per-dir). Typical Maven
directories hold 5â€“20 files; Pantera's "huge directory" case is
theoretical. If a pathological directory emerges, the next evolution is
keyset pagination in the handler (the response already advertises
`marker` / `hasMore`).

## Non-goals / follow-ups

- `?include=metadata,checksums,â€¦` on `/api/v1/search` â€” structurally
  plumbed via field filters; query-param plumbing deferred.
- Gradle adapter event emission â€” not done; the module has no source
  code and is not in the reactor (it's a stale leftover). Documented
  as dead.
- Deleting `FileSystemBrowseSlice` â€” still wired as the HTML browsing
  fallback for non-UI clients; scope to revisit after the JSON tree
  UX beds in.

## Tests

All regression tests run in `DbArtifactIndexTest` and
`ArtifactHandlerTest` against a Testcontainers PostgreSQL with the
full Flyway migration set applied (via a new `DbManager.migrate()`
call in the test setUp; previously only the core schema was built via
`ArtifactDbFactory.createStructure()` and tests couldn't see any
Flyway-managed columns including `version_sort`).

| Test | Covers |
|------|--------|
| `createdAtSortHonoursDirection` | Bug A / RC1 |
| `toSortFieldRejectsEnumNameFormForDate` | Contract pin: `"date"` is not a wire key |
| `nameSortIsNatural` | Bug C / RC3 (natural name order) |
| `searchExcludesNonArtifactKinds` | Bug B / RC2 (noise filter) |
| `classificationFunctionLabelsKnownPatterns` | 19-case contract for `classify_artifact` |
| `treeEchoesSortParams` | Bug E default / param round-trip / fallback |

Good-day / bad-day / edge coverage:
- **Good day:** asc/desc returns correctly ordered distinct items;
  tree listing surfaces dates; metadata filter hides 6 sidecar rows.
- **Bad day:** DB hydration failure in `treeHandler` returns the plain
  storage listing instead of 5xx (exercised via log observation).
- **Edge:** unknown `sort=garbage` degrades to `name`; `"date"` wire
  value does not resolve to `SortField.DATE` (contract pin);
  `maven-metadata.xml.md5` classifies as `CHECKSUM` because the
  checksum rule fires first (both kinds are user-invisible, so the
  observable behaviour is identical).
